¶page table

  1. address translation

    RISC-V instructions manipulate virtual addresses; RAM, or physical memory is indexed with physical address. The RISC-V page table hardware provides mapping of virtual address to physical address.

  2. kernel address space:

    • Xv6 maintains one page table per process(user address space) and single page table for the kernel address space.

    • Kernel uses direct mapping to get access to RAM and memory mapped devices.

    • Not direct mapped: high-memory mapping

      • trampoline page

      • kernel stack pages: have an unmapped guard page with PTE invalid.

  3. user address space

  4. process of vm setup

    • physical memory allocation: free pages are stored in the linked list.
    • Syscall
      • sbrk: grow/shrink the memory
      • exec: create the user part of an address space

¶interrupts

  1. Drivers’s data structures can divide into two parts, top/bottom.

    • Top: user process or interface of kernel’s other part’s call, like read/write()

    • Bottom: interrupt handler

  2. Driver is programmed by memory mapped I/O.

  3. How xv6 displays “$” in console?

    device -> (uart->interrupt ->) console

  4. How xv6 displays “ls” in console?

    key stroke -> keyboard -> (uart input -> merge to byte -> interrupt)->interrupt handler

  5. Riscv interrupts related registers:

    • SIE(supervisor interrupt enable),
    • SSTATUS(supervisor status)
    • SIP(supervisor interrupt pending)
    • SCAUSE: means interrupts
    • STVEC: program counter of user program that cpu runs when trap/interrupts/page faults occur

¶Parallelism

¶lock

  1. Xv6 has two locks: spinlocks and sleeplocks

    • spinlocks
      • disable interrupts: if a lock can be interruptted, then the interrupt acquires the same lock that is being used which leads to dead lock. -> concurrency in same CPU
      • sw vs atomic_swap: sw differs in different CPU settings. It may have one more instruction which disables the atomicity. -> concurrency in different CPU
    • Sleeplocks -> sleep/wakeup?

  2. Why locks: one data stucture are accessed by multiple processes(race condition) -> atomicity in critical section.

  3. How? system designers should define orders of locks for the purpose of performance and correctness

Scenario 1: race condition in kfree()

Attempt 1: corse-grain locking/two processs are accessing one sharing data structure and one of them updates the data.

scenario 2: rename files: rename(“d1/x”, “d2/y”)

Attempt 2: lock operations not data structures.

Terminology:

  1. Deadlock: unlocking A requires B, unlocking B requires A, A and B are locked. example: deadly embrace

¶threads

  1. Why threads? Run parallel programs

  2. What: One thread is the state of one serial execution. Thread states include PC, REGS, STACK, [state:runnning, runable, sleeping]

  3. How: problems of implementing threads

    Assumption: one core(CPU) one thing -> thread either sleeps or runs on one core

    • how to switch between processes(scheduling)
    • What to save and restore in switching
    • Compute-bound threads: long-running program

Problem1: what if the process runs for a long time?

-> [Pre-emptive scheduling] the process is:

  1. user processes run
  2. time interrupts make it give up control to kernel
  3. kernel then yields the CPU

while cooperative scheduling is user processes explicitly give up control to kernel, like QEMU coroutines.

¶File systems

¶xv6 design

Xv6 fs addresses four problems of implementing fs:

  1. fs structure: dir -> file
  2. crash recovery: logging
  3. coordination: invariants of a fs?
  4. in-memory cache

Layout of xv6 fs:

||Boot block | super block(mkfs) | log(latest transaction to restore if crushes) | inode(index, block offset) | bit map(blocks in use) || data ||

kernel/bio.c: Xv6 fs uses buffer caches to store N least recently used blocks of data. Doubly-linked lists are used as buffers.

problem: Why <= 1 cached buffer per disk sector?

It means one buffer can only store one sector’s data. If sector x is stored in two buffers, imagine read(sector x): when reading, the data in sector x must be in sync with buffers written to the sector. Now think about which approach is better for handling synchronization?

  • approach 1: read x -> (buffer 1, buffer 2 -> )sector x
  • approach 2: read x -> buffer 0 -> sector x
